# Frontend Display Rework Plan

## 补充决策（2026-06-25）

- 首页“通道”列不再显示 `new-api` 原始渠道名。
- 首页“通道”列主文本统一显示审计通道类型：
  - `官方直连`
  - `混合`
  - `逆向`
  - `未知`
- 原始渠道名只保留在 tooltip 中，作为辅助信息。

## 1. 目标与最终产物

本次前端改造只围绕两件事执行：

1. 首页 `/` 作为入口页，展示 `new-api` 同步过来的渠道列表
2. 点击首页服务商 / 渠道后，进入一个**模型级详情页**

最终产物必须满足：

- 首页主数据来自 `new-api` 渠道同步结果
- 首页只显示入口决策需要的信息：
  - 服务商
  - 服务
  - 通道
  - 可用率
  - 可用率趋势
- 首页不显示模型
- 详情页不是旧的通道级监控列表，而是**按模型展开的质量排名页**
- 详情页结构、字段顺序、说明区块对齐用户给出的截图

本次明确废弃：

- `/audit`
- `/audit/ranking`
- 旧版 `ProviderPage` 的通道级主视图
- “先做一个差不多版本再说”的思路

## 2. 当前错误点

当前实现有 4 个核心错误：

1. 首页虽然已切到 `new-api` 渠道快照，但详情页仍沿用旧 `ProviderPage`，主体还是通道级状态表。
2. 用户要求的是“进入后显示每个模型的状态”，当前页面却仍然显示“每个通道一行”。
3. 当前详情页没有落下截图中的标题区、说明条、图例区、模型级排名表。
4. 当前详情页的列结构不对，缺少模型级的最终质量分、机器指纹分、测试数、30D 指标等。

因此，后续执行必须从“修补旧详情页”切换为“重建详情页的数据视图”，但仍优先复用现有样式、控件和局部组件。

## 3. 页面边界

### 3.1 首页 `/`

首页职责固定为入口页：

- 数据源：`/api/audit/channels`
- 粒度：渠道级
- 用途：让用户选中某个服务商 / 通道进入详情

首页字段固定为：

- 服务商
- 服务
- 通道
- 可用率
- 可用率趋势

首页禁止显示：

- 模型
- 价格
- 收录
- 最后监测
- 质量列

### 3.2 详情页 `/p/:provider?service=...&channel=...`

详情页职责固定为模型级明细页：

- 主对象：选中的 `服务商 + 服务 + 通道`
- 粒度：模型级
- 一行代表一个模型
- 用途：看该通道下每个模型的真实性、质量、可用率和趋势

## 4. 数据边界

### 4.1 主数据源

详情页主对象必须来自 `new-api` 同步结果：

- 服务商
- 服务
- 通道
- 模型列表
- 启停状态

已确认当前可用来源：

- `/api/audit/channels`

该接口已能提供：

- `provider`
- `service`
- `channel`
- `model`
- `enabled`

其中 `model` 是逗号分隔模型串，需要在详情页拆成模型级行。

### 4.2 补充数据源

模型级页需要补充使用现有监控 / 质量数据：

- `useMonitorData`：通道级可用率、趋势、最后检测、延迟等
- `useRpdiagScores`：模型级质量分、趋势、指纹分、详情链接

补充规则：

- `new-api` 渠道同步结果负责定义“有哪些模型”
- `rpdiag` 负责补充“每个模型的质量分 / 趋势 / 指纹相关信息”
- `monitorData` 负责补充“通道级可用率 / 趋势 / 延迟”
- 如果某个模型没有质量分，也必须显示出来，状态为“不可测/暂无数据”，不能直接消失

## 5. 详情页目标结构

详情页必须按截图落为 5 个区块。

### 5.1 标题区

必须包含：

- 标题：`服务商质量排名`
- 副标题：`基于协议指纹与官方近 3 次基线的加权相似度评分（0-100），分数越高代表与官方行为越接近。`
- 注释条：`仅供参考的统计观察；权重见方法论，不构成定性结论。`

### 5.2 服务切换

必须提供服务切换：

- `Claude Code`
- `Codex`

要求：

- 不是重新造新路由
- 直接绑定现有 `service` 过滤语义
- 切换后联动筛选区和模型表

### 5.3 筛选区

详情页最少保留这 4 个筛选器：

- 来源
- 服务商
- 通道
- 模型

约束：

- 默认进入详情页时，服务商和通道自动带入当前入口对象
- 模型筛选器基于当前通道下的模型集合生成
- 来源筛选优先从通道命名规范 / rpdiag key / raw 字段推导

### 5.4 图例区

必须有两条说明条。

第一条：通道符号说明

- 董推 / 推荐
- rpdiag 直连基准
- 服务商自报官方通道 (O-)
- 逆向 (R-)
- 混合 (M-)
- 未知
- 用户提交 (U-)

第二条：分数含义说明

- `≥85`：与官方近 3 次基线高度接近
- `70-84`：多数维度接近基线
- `50-69`：与基线存在可见偏离
- `<50`：与基线偏离较明显

### 5.5 模型级排名表

一行一个模型，列顺序按截图执行：

1. `#`
2. `服务商`
3. `通道`
4. `模型`
5. `最终质量分`
6. `机器指纹分`
7. `趋势`
8. `测试数`
9. `可用率 30D`
10. `首 TOKEN 30D`
11. `平均延迟 30D`

字段落地规则：

- `服务商`：当前 provider
- `通道`：当前 channel
- `模型`：由 `new-api` 同步模型列表拆分生成
- `最终质量分`：优先取 rpdiag 的模型 score
- `机器指纹分`：先映射为 rpdiag 的模型 score / max_score 派生字段；若当前 wire 不足，先保留占位但列必须存在
- `趋势`：优先展示 rpdiag 模型级 sparkline / dot
- `测试数`：优先取 rpdiag `n_30d` 或可推导计数
- `可用率 30D`：先复用当前 monitor 可用率，若当前仅有通道级则先按通道级补齐并明确口径
- `首 TOKEN 30D`：先复用/映射现有延迟或 TTFT 口径，没有真实值时显示占位
- `平均延迟 30D`：先复用现有 latency 口径，没有真实值时显示占位

## 6. 组件复用策略

优先复用，不再平行造轮子：

- `Header`
- `Controls`
- `Tooltip`
- `StatusDot`
- `ChannelTypeIcon`
- `useAuditChannels`
- `useMonitorData`
- `useRpdiagScores`

允许新增但必须最小化的内容：

- 模型级详情页的行数据适配器
- 模型级详情表组件
- 详情页专用说明条 / 图例条

禁止事项：

- 不再把 `StatusTable` 强行当成模型级详情表继续套
- 不再把旧 `ProviderPage` 视为最终结构，只能把它当容器壳
- 不新增无关页面

## 7. 实施顺序

### 第一步：重构详情页数据视图

- 从 `/api/audit/channels` 中定位当前 `provider + service + channel`
- 将 `model` 字段拆分成模型数组
- 构造模型级基础行

### 第二步：补齐模型级质量数据

- 用 `useRpdiagScores` 按 `provider + service + channel` 取质量对象
- 将 `rpdiag.models[]` 与 `new-api` 模型列表做归并
- 产出每个模型的：
  - score
  - trend
  - detail_url
  - 测试计数
  - 故障提示

### 第三步：补齐通道级可用率与趋势

- 从 `useMonitorData` / 现有匹配索引补：
  - 可用率
  - 趋势热力块
  - 延迟
  - 最后检测

说明：

- 当前如果没有模型级 availability，只允许先落“通道级补充到模型行”
- 但 UI 上必须保持一行一个模型

### 第四步：重做详情页结构

- 替换旧 `ProviderPage` 主体
- 新增：
  - 标题区
  - 服务切换
  - 筛选区
  - 图例区
  - 模型级排名表

### 第五步：清理错误实现

- 删除本次目标无关的旧详情页主视图逻辑
- 删除用户已判定无效的临时前端表达
- 保留首页入口页与现有公共组件

## 8. 验收标准

本次验收只看 6 条：

1. 首页仍然是入口页，且不显示模型
2. 首页点击后进入详情页，不进入旧通道级列表
3. 详情页标题区、注释条、服务切换、筛选区、图例区都存在
4. 详情页表格是一行一个模型，不是一行一个通道
5. 模型表列顺序与截图一致
6. `localhost:8080` 运行态可直接看到结果

## 9. 执行结论

接下来不再继续修补当前错误详情页。

后续执行方向固定为：

- 首页保留渠道入口职责
- 详情页改为模型级质量明细页
- 数据主对象仍来自 `new-api` 同步结果
- 监控与 rpdiag 只作为补充指标层
